# Phase 1: 识别意图（Intent Recognition）

## 目标

将模糊的自然语言请求转化为清晰、可执行的任务定义。

## 核心问题清单

收到任务后，必须逐一回答以下问题：

### 1. 任务类型

| 类型 | 示例 |
|------|------|
| ✅ 新建功能（New Feature） | "添加用户注册功能" |
| ✅ 修改/重构（Refactor） | "把支付模块抽出来" |
| ✅ 修复 Bug（Bug Fix） | "登录按钮点了没反应" |
| ✅ 添加测试（Testing） | "给订单模块加测试" |
| ✅ 部署/配置（DevOps/Config） | "把服务部署到 K8s" |
| ✅ 性能优化（Performance） | "首页加载太慢了" |

### 2. 复杂度评估

| 级别 | 标准 | 行动 |
|------|------|------|
| **Trivial** | 1-3 行改动，逻辑一目了然 | **跳过流程**，直接做 |
| **Small** | 单文件改 < 10 行 | 走简化流程 |
| **Medium** | 多文件，需规划 | **走完整 6 步** |
| **Large** | 跨多个子系统 | 走完整流程 + 子代理并行 |
| **X-Large** | 需设计评审 + 多轮迭代 | 走完整流程 + Workflow 编排 |

### 3. 关键约束

- **涉及哪些文件/目录？**
- **技术栈/语言/框架？**
- **有没有已有的代码模式要遵循？**
- **性能/安全/兼容性要求？**
- **截止时间/优先级？**

## 输出

识别意图完成后，应输出：

```markdown
## 任务定义

- **类型**: New Feature
- **复杂度**: Medium（3-5 个文件）
- **涉及文件**: src/routes/auth.ts, src/models/user.ts
- **技术栈**: Node.js + Express + Prisma + JWT
- **约束**: 密码需 bcrypt 加盐，Token 有效期 24h
```

## 常见错误

❌ **跳过了意图识别** — 用户说"加个功能"，
直接开写，写到一半发现理解错了
✅ **正确做法**：先问清楚"具体什么功能？接受什么输入？输出什么？"

❌ **复杂度评估错误** — 把 Large 判成 Small，
结果写到一半发现需要改 10 个文件
✅ **正确做法**：先 grep 搜索相关文件，确认影响范围再评估

❌ **忽略隐式约束** — 用户说"加条路由"，
没问安全性，结果路由没有鉴权
✅ **正确做法**：主动问"需要鉴权吗？需要对特定角色开放吗？"
