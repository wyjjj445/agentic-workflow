# Phase 5: 干活（Execute）

## 目标

审查通过后，高效、安全地执行代码修改。

## 核心原则

### 1. Read before Edit

**任何文件在被修改之前，都必须先读取。**

```
✅ 正确做法：
  Read("src/routes/users.ts")     ← 先读
  理解内容后
  Edit("src/routes/users.ts", ...) ← 再改

❌ 错误做法：
  Edit("src/routes/users.ts", ...) ← 直接改，不知道原来有什么
  → 失败：内容不匹配
```

### 2. 一次改完

相关的改动在同一个上下文中完成。不要改一半去做别的事。

```
✅ "添加用户管理功能":
  Read(模型) → Edit(模型)
  Read(路由) → Edit(路由)
  Read(测试) → Edit(测试)
  测试通过后 → 一起交付

❌ 先改模型，去做别的事，半天后回来改路由 → 上下文已丢失
```

### 3. 只改动必要的行

- **不要格式化无关代码** — 只改你需要改的行
- **不要做"顺手"的重构** — 引入不相关变更会混淆审查
- **保持 diff 最小** — 越小越容易审查，越不容易引入 bug

### 4. 改完立即验证

```
Write/Edit → 运行测试 → PASS → 继续下一步
                 ↓ FAIL
             分析根因 → 修复 → 重测
```

## 迭代循环

每个 Task 的执行都是一个小循环：

```
      ┌──────────┐
      │  写代码    │
      └────┬─────┘
           │
           ▼
      ┌──────────┐      FAIL      ┌──────────┐
      │  跑测试    │ ─────────>   │  修复     │
      └────┬─────┘               └────┬─────┘
           │                          │
         PASS                        │
           │                          │
           ▼                          │
      ┌──────────┐                   │
      │  Review   │                  │
      │  审查通过   │                  │
      └────┬─────┘                   │
           │                          │
         DONE ◄──────────────────────┘
```

## 调试技巧

### 测试失败后分析根因

```
测试失败 → 不要直接重试

Step 1: 看错误信息
  → "Expected 200, got 500"

Step 2: 分析可能原因
  → 路由没注册？
  → 中间件报错？
  → 数据库连接失败？

Step 3: 针对性检查
  → Read("src/routes/index.ts") → 发现路由没注册

Step 4: 修复 → 重测
```

### 边改边测

不要等全部写完了再测，改一点测一点：

```
Edit("user.model.ts") → 跑 user.test.ts → PASS
Edit("auth.route.ts") → 跑 auth.test.ts → PASS
Edit("user.service.ts") → 跑 user.test.ts → PASS（回归测试）
```

## 禁止行为

- ❌ 不 Read 直接 Edit
- ❌ 改了代码不跑测试
- ❌ 格式化/重构无关代码
- ❌ 测试失败直接重试（不分析根因）
