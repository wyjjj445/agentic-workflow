# Phase 4: Review（多维度审查）

## 目标

在合并代码前，从多个维度进行全面审查，确保代码质量。

## 审查维度

```
                     ┌──────────────────────┐
                     │  1. Correctness       │
                     │    逻辑是否正确？      │
                     ├──────────────────────┤
                     │  2. Efficiency        │
                     │    O(n)？缓存？        │
                     ├──────────────────────┤
                     │  3. Simplification    │
                     │    能否更简洁？        │
                     ├──────────────────────┤
                     │  4. Test Coverage     │
                     │    边界条件覆盖？      │
                     ├──────────────────────┤
                     │  5. Security          │
                     │    XSS/SQL注入/权限？  │
                     ├──────────────────────┤
                     │  6. Error Handling    │
                     │    异常路径处理？      │
                     └──────────────────────┘
```

## 审查清单

### 1. Correctness（正确性）

- [ ] 逻辑是否完整？所有分支都覆盖了吗？
- [ ] 边界条件：空值、极限值、特殊字符？
- [ ] 并发安全性？竞态条件？
- [ ] 状态变化是否和预期一致？
- [ ] 类型是否正确？

### 2. Efficiency（性能）

- [ ] 时间复杂度？（需要优化吗？O(n²) 变成 O(n log n)？）
- [ ] 是否有不必要的重复计算？可以加缓存吗？
- [ ] 是否有 N+1 查询问题？
- [ ] 是否有内存泄漏可能（事件监听器未解绑）？

### 3. Simplification（简洁性）

- [ ] 这段代码能否更短/更清晰？
- [ ] 有重复代码可以提取方法吗？
- [ ] 命名是否 self-documenting？
- [ ] 有没有多余的抽象？（过度设计）
- [ ] 注释是否必要？是否在解释"为什么"而非"是什么"？

### 4. Test Coverage（测试覆盖）

- [ ] 核心逻辑有测试吗？
- [ ] 边界条件测试了哪些？
- [ ] 错误路径有测试吗？
- [ ] 测试本身是否清晰、易维护？

### 5. Security（安全）

- [ ] 输入验证充分吗？（XSS、SQL 注入、命令注入）
- [ ] 鉴权/授权检查了吗？
- [ ] 敏感数据加密了吗？
- [ ] 有速率限制/Risk 控制吗？

### 6. Error Handling（错误处理）

- [ ] 所有错误都被捕获了吗？
- [ ] 错误信息对用户友好吗？
- [ ] 不会泄漏内部实现细节（如堆栈信息）？
- [ ] 失败时有回滚/补偿机制吗？

## 执行方式

### 1. 手动审查

```
// 遍历所有修改过的文件，逐维度检查
文件: src/routes/auth.ts
  └─ Correctness: 密码验证逻辑正确 ✅
  └─ Efficiency: 每次请求都查库，可加 Redis 缓存 ⚠️
  └─ Security: 使用了 bcrypt ✅
  └─ Error Handling: 登录失败没有区分"用户不存在"和"密码错误" ⚠️
```

### 2. 使用 /code-review

```
/code-review [--low|--medium|--high]
```
- `--low`: 快速扫描，仅高置信度问题
- `--medium`: 平衡精度与覆盖率
- `--high`: 深入审查，可能包含不确定的发现

### 3. 对抗验证（Adversarial Verify）

对关键代码（鉴权、支付、安全逻辑），使用对抗验证：

```
Step 1: 派发 N 个子代理
  子代理 1: "反驳这段代码有 bug" → 发现密码验证逻辑可能绕过
  子代理 2: "反驳这段代码有 bug" → 找不到问题
  子代理 3: "反驳这段代码有 bug" → 验证通过

Step 2: 投票（2/3 通过）
  → 2/3 反对（认为没问题）→ 可信 ✅
  → 2/3 赞成（认为有问题）→ 退回调修 ❌
```

## 常见陷阱

❌ **只审查了自己的代码** — 子代理写的代码也要审查
❌ **只关注正确性** — 忽略了安全和错误处理
❌ **发现问题但没有修复流程** — 发现 Bug 后应该创建修复 Task 追踪，而不是口头提到
❌ **假设测试会覆盖一切** — 测试没写等于没覆盖
