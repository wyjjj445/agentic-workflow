# Phase 6: 子代理测试（SubAgent Test / Verify）

## 目标

端到端验证功能**确实**按预期工作，而不是只看单元测试通过。

> "单元测试都过了" ≠ "功能正常工作"

## 验证层级

```
                    ┌───────────────────────┐
                    │  Level 1: Unit Tests  │  ← Phase 5 已做
                    │  单元测试              │
                    ├───────────────────────┤
                    │  Level 2: Integration │
                    │  集成测试              │
                    ├───────────────────────┤
                    │  Level 3: E2E         │
                    │  端到端验证            │  ← Phase 6 核心
                    ├───────────────────────┤
                    │  Level 4: User Flow   │
                    │  用户流程验证          │
                    └───────────────────────┘
```

## 验证方法

### 1. 使用 verify 子代理

```javascript
Skill({skill: "verify"})

// 或者手动调用
Agent({
  subagent_type: "general-purpose",
  prompt: "验证新实现的登录功能是否能正常工作",
})
```

verify 子代理会：
1. 运行项目测试
2. 手动模拟用户使用场景
3. 检查边界条件
4. 检查错误路径
5. 报告验证结果

### 2. 运行项目测试

```bash
# Node.js
npm test

# Python
pytest

# Rust
cargo test

# Go
go test ./...

# Java
./mvnw test
```

### 3. 手动场景模拟

```javascript
// 验证注册流程
1. POST /api/register -> 返回 201 + token
2. POST /api/register (重复) -> 返回 409 Conflict
3. POST /api/register (空密码) -> 返回 400 Bad Request
4. POST /api/login -> 返回 200 + token
5. POST /api/login (错误密码) -> 返回 401 Unauthorized
6. GET /api/profile (带 token) -> 返回 200 + 用户信息
7. GET /api/profile (无 token) -> 返回 401
```

### 4. 回归验证

验证**新功能没有破坏旧功能**：

```
新增: 用户注册功能
回归: 
  - 原有登录功能是否正常？✅
  - 原有用户列表是否正常？✅
  - 原有鉴权中间件是否正常？✅
```

## 验证通过标准

### ✅ 必须满足

- [ ] 所有测试通过
- [ ] 核心功能路径正常工作
- [ ] 错误路径有合理处理

### ⚠️ 尽量满足

- [ ] 边界条件覆盖（空值、极限值、特殊字符）
- [ ] 并发场景基本正常
- [ ] 性能没有明显回退

### ❌ 不算通过

- 测试失败但"看起来问题不大"
- 只测了 happy path，没测错误路径
- 忘了回归测试旧功能
- 验证了但选择性地忽略失败的测试

## 失败处理

```
验证失败
  │
  ├─ 回到 Phase 5 修复
  │   ├─ 分析根因
  │   ├─ 修改代码
  │   └─ 跑测试
  │
  └─ 重新 Phase 6 验证
      └─ 通过 → 完成
```

## 常见错误

❌ **只看单元测试** — 单元测试通过不代表集成后能工作
❌ **忘记回归** — 新功能好了，旧功能坏了
❌ **选择性忽略失败** — "这条失败是因为环境问题，不用管"
✅ **正确做法**：真实验证每个 use case，无论失败大小都修复
